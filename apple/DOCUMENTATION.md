- [2026-06-22 20:56]: Crash fix — enabling iCloud sync instantly crashes (`EXC_BREAKPOINT` in CloudKit)
  - *Details*: The macOS App Store build (1.0.5/6) crashed the instant the user toggled iCloud sync on. Symbolicating the crash against this machine's identical CloudKit (2360.120.2, macOS 26.5.1/25F80) recovered the precondition: **"BUG IN CLIENT OF CLOUDKIT: Cannot await a call into CKSyncEngine from within a delegate callback if that function will end up calling back into the delegate."** (`CloudKit/CKSyncEngine.swift:286`). Root cause: when a `CKSyncEngineDelegate` event handler throws, `CloudKitSyncService.handleEvent(_:syncEngine:)`'s `catch` calls `stopAfterFatalError(_:)`, which did `await existingEngine.cancelOperations()` **while still on CloudKit's delegate-callback task**. CloudKit marks "inside a callback" with a task-local; awaiting a re-entrant engine call there violates its serial-callback guarantee, so it traps. The `catch`-then-teardown is the only path in the service that awaits back into the engine from a callback (verified: `start()`/`stop()`/`synchronize()`'s engine awaits all run on non-callback tasks; the synchronous `engine.state.add/remove` mutations in callbacks are allowed; `RecordZoneChangeBatch(pendingChanges:recordProvider:)` is the sanctioned in-callback API and does not call back into the *delegate*).
  - *Tech Notes*: One-line fix in `Sources/Shared/Services/CloudKitSyncService.swift` (`stopAfterFatalError`) — `await existingEngine.cancelOperations()` → `Task.detached { await existingEngine.cancelOperations() }`. **Must be `Task.detached`, not `Task { }`**: an unstructured `Task { }` inherits the enclosing task-locals (CloudKit's "in-callback" marker included) and would still trap; `Task.detached` starts with a clean task-local context — exactly what the precondition message recommends ("Try performing this in a detached Task"). `engine` is already set to `nil` before the detach, so callbacks that fire before cancellation lands are ignored by the existing `engine === syncEngine` guards. No API/signature changes. **Cross-platform:** `CloudKitSyncService` lives in `Sources/Shared/` and compiles into BOTH apps, so iOS had the identical latent crash (same `handleEvent` catch → `cancelOperations` path; iOS reaches it via `ContentView.refreshICloudSyncIfNeeded()` / the sync toggle just like macOS) — this fix covers both. Could not run a full `xcodebuild` here (no Xcode on this box, CommandLineTools only); change is self-contained and uses the existing `Task` pattern already present in `FinanceStore`. *Open follow-up (not done here):* `handleEvent` still treats any thrown error as fatal and tears down the engine — see `WealthCompass/TO_IMPROVE.md` #6 (skip a single bad record instead of throwing) to keep sync from stopping after a transient/per-record error. Post-fix the worst case is a non-crashing "Sync Error", not a crash.
