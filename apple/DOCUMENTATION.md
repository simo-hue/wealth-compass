- [2026-07-06]: Deep-audit ‚ÄĒ Sync trio (L37, L39, L52) ‚ÄĒ ‚Ź≥ pending **on-device sync verification** (two devices / one iCloud account)
  - *Details*: On `low-sync-trio` off `main`. Deep CloudKit-actor concurrency work; adversarially reviewed. These are the "confirm on-device" items ‚ÄĒ no local Xcode, and the concurrency/init-timing paths aren't cleanly unit-testable without a live `CKSyncEngine`.
  - *Tech Notes*:
    - **L37 ‚ÄĒ investigated, deliberately NOT reordered (comment only).** The audit proposed "persist, then commit `cached`" in `CloudSyncMetadataStore.update()`. A careful read shows `update()` runs concurrently (the actor's sync handlers AND the MainActor via `reconcileLocalInventory`/`recordLocalChanges`), and `update()`/`reset()` both lock in dataLock‚ÜíwriteLock order. Persist-then-commit forces either (a) re-acquiring `dataLock` while holding `writeLock` ‚Üí an AB‚ÜĒBA **deadlock** with `reset()`/a concurrent `update()`, or (b) releasing `dataLock` before commit ‚Üí a concurrent `update()` reads the pre-commit value and **loses an update**. So commit-before-persist is the correct, deadlock-safe, lost-update-safe choice; the residual harm (a persist failure immediately followed by a process kill leaves disk one advance behind memory) is bounded, idempotent, self-correcting (redundant re-sync on next launch, never data loss). Added a detailed comment so nobody "fixes" it into a deadlock later.
    - **L39** ‚ÄĒ strengthened the post-`await` reconcile guard in `handleFetchedRecordZoneChanges`: it also requires the tombstone state to be unchanged since the plan was built (`currentTombstone == originalTombstone`). This closes the `nil == nil` pending-revision window where a stale pre-`await` plan (isTombstone=false) could overwrite a tombstone a reentrant fetched batch applied during the `await`, resurrecting a just-deleted record. Purely defensive ‚ÄĒ a differing state only routes into the existing conservative re-queue branch, never resurrects; a normal (non-racing) apply always has matching tombstone state.
    - **L52** ‚ÄĒ the init-time "restore sync-enabled" `Task` is now tracked in `initialSyncEnableTask`, cancelled in `deinit` and at the top of `wipeLocalState()` (before `stop()`), so a factory reset mid-launch can't let it restart the engine after the wipe. Added a defensive `guard isICloudSyncEnabledResolved` for the automatic (`userInitiated == false`) path in `setICloudSyncEnabled` so a late start after the flag was cleared is a no-op. The guard is scoped to the automatic path only (the two Settings toggles are `userInitiated: true` and unaffected).
