- [2026-06-22 20:56]: Crash fix — enabling iCloud sync instantly crashes (`EXC_BREAKPOINT` in CloudKit)
  - *Details*: The macOS App Store build (1.0.5/6) crashed the instant the user toggled iCloud sync on. Symbolicating the crash against this machine's identical CloudKit (2360.120.2, macOS 26.5.1/25F80) recovered the precondition: **"BUG IN CLIENT OF CLOUDKIT: Cannot await a call into CKSyncEngine from within a delegate callback if that function will end up calling back into the delegate."** (`CloudKit/CKSyncEngine.swift:286`). Root cause: when a `CKSyncEngineDelegate` event handler throws, `CloudKitSyncService.handleEvent(_:syncEngine:)`'s `catch` calls `stopAfterFatalError(_:)`, which did `await existingEngine.cancelOperations()` **while still on CloudKit's delegate-callback task**. CloudKit marks "inside a callback" with a task-local; awaiting a re-entrant engine call there violates its serial-callback guarantee, so it traps. The `catch`-then-teardown is the only path in the service that awaits back into the engine from a callback (verified: `start()`/`stop()`/`synchronize()`'s engine awaits all run on non-callback tasks; the synchronous `engine.state.add/remove` mutations in callbacks are allowed; `RecordZoneChangeBatch(pendingChanges:recordProvider:)` is the sanctioned in-callback API and does not call back into the *delegate*).
  - *Tech Notes*: One-line fix in `Sources/Shared/Services/CloudKitSyncService.swift` (`stopAfterFatalError`) — `await existingEngine.cancelOperations()` → `Task.detached { await existingEngine.cancelOperations() }`. **Must be `Task.detached`, not `Task { }`**: an unstructured `Task { }` inherits the enclosing task-locals (CloudKit's "in-callback" marker included) and would still trap; `Task.detached` starts with a clean task-local context — exactly what the precondition message recommends ("Try performing this in a detached Task"). `engine` is already set to `nil` before the detach, so callbacks that fire before cancellation lands are ignored by the existing `engine === syncEngine` guards. No API/signature changes. **Cross-platform:** `CloudKitSyncService` lives in `Sources/Shared/` and compiles into BOTH apps, so iOS had the identical latent crash (same `handleEvent` catch → `cancelOperations` path; iOS reaches it via `ContentView.refreshICloudSyncIfNeeded()` / the sync toggle just like macOS) — this fix covers both. Could not run a full `xcodebuild` here (no Xcode on this box, CommandLineTools only); change is self-contained and uses the existing `Task` pattern already present in `FinanceStore`. *Open follow-up (not done here):* `handleEvent` still treats any thrown error as fatal and tears down the engine — see `WealthCompass/TO_IMPROVE.md` #6 (skip a single bad record instead of throwing) to keep sync from stopping after a transient/per-record error. Post-fix the worst case is a non-crashing "Sync Error", not a crash.

- [2026-06-23]: iCloud sync — split foreground sync API + debounce (TO_IMPROVE.md #7)
  - *Details*: App activation (`ContentView`/`MacRootView` `handleAppBecameActive`) called `finance.refreshICloudSyncIfNeeded()` → `cloudSyncService.start(...)` on every foreground. `start()` already early-returned to `synchronize()` when the engine existed (so it didn't re-run the heavy startup), but it meant **every** return-to-foreground fired a full un-debounced fetch+send. Split the "ensure running" and "sync now" concerns and debounced the opportunistic path.
  - *Tech Notes*:
    - **`CloudKitSyncService`** — `start(allowAccountReplacement:)` is now purely "ensure running": when an engine already exists it returns immediately (was: fall through to `synchronize()`); it still does the full startup (account check → user-record → inventory reconcile → engine setup → one initial sync) only when `engine == nil`. New `requestSync()` is sync-only: it no-ops when the engine isn't running (it never starts it) and is debounced by a new in-memory `lastSyncStartedAt` against `foregroundSyncMinimumInterval = 30s`. `synchronize(generation:using:)` stamps `lastSyncStartedAt` the moment a sync commits to running, so cold-start, Force Sync, and change-driven syncs all reset the window. Force Sync (`synchronize()`) and change-driven `localChangesRecorded()` are intentionally **not** debounced. Kept the method name `start` (one existing test calls it; no rename churn).
    - **`FinanceStore`** — replaced `refreshICloudSyncIfNeeded()` with `ensureICloudSyncRunning()` (→ `start`) and `requestICloudSync()` (→ `requestSync`); extracted the enabled-check into a private `isICloudSyncEnabledResolved` (settings, falling back to the persisted `wc_mobile_icloud_sync_enabled` default), now also used by `forceICloudSync`. They change sync **semantics**, so they need a real two-device / cross-version iCloud test (a clean build is necessary but not sufficient) — runtime checklist in `TO_SIMO_DO.md` #28–29. **No CloudKit schema or JSON wire-format change.** No Xcode here (CommandLineTools only) — not built/run by me.
  - *Tech Notes*:
    - **WC-H3 (`372e461`) — one undecodable remote record no longer tears down the engine.** `FinancialData.applyCloudSyncMutations` now applies each remote mutation under a per-record `do/catch` and returns a `RemoteMutationOutcome` (`appliedKeys` + `skipped`); a decode failure or id mismatch is skipped+logged instead of thrown, so it no longer propagates through `handleFetchedRecordZoneChanges` → `handleEvent`'s catch → `stopAfterFatalError` (which permanently disabled, then re-poisoned, sync). `applyRemoteMutations` logs skips via OSLog and returns only the applied keys — so a skipped record's `knownLocalHashes` is never advanced. That last point is the sharp edge: advancing it would make the next `reconcileLocalInventory` see the key as locally-deleted and enqueue a tombstone, i.e. an old app version would *delete a record it merely couldn't decode*. Mirrors the existing payloadless `remoteSnapshot(...)` nil-skip; keeps the pendingRevision/applicable-mutations filter intact. The sent-conflict path is left untouched: an undecodable server-wins record falls through to the existing `conflictAction(server: nil) → requeueLocal` policy (same as a payloadless server record).
    - **WC-H4 (`fcfe7b9`) — encode the snapshot once per send batch, not once per record.** `nextRecordZoneChangeBatch` now builds the batch via a new `makeRecords(for:)` that calls `snapshotProvider()` once and indexes the result; `makeRecord` takes the snapshot as a parameter (no per-record provider call) and the engine/lifecycle guard stays in `nextRecordZoneChangeBatch`, leaving `makeRecords` engine-free + unit-testable. Was B×N full-dataset JSON-encode+SHA-256 on the main actor per batch; now one encode + one main-actor hop. Also memoized `cloudSyncRecords()` in `FinanceStore` keyed by the existing `dataVersion` (`currentCloudSyncRecords()` + `cachedCloudSyncSnapshot`).
    - **WC-M3 (`d2d6c2e`) — metadata store no longer holds a lock across disk writes.** `CloudSyncMetadataStore` split into `dataLock` (guards `cached`, used by `read()`) and `writeLock` (serializes the full-file atomic write). `update` mutates `cached` under `dataLock`, takes `writeLock` *before* releasing `dataLock` (hand-over-hand), then persists outside the `dataLock` critical section — so `read()` (the hot actor path) never blocks on disk and writes stay ordered. Stays synchronous + throwing (so WC-M2 can classify a persist failure). `reset` follows suit. On-disk format unchanged.
    - **WC-M2 (`24030ef`) — transient errors no longer self-disable sync.** `handleEvent`'s catch tears the engine down (and disables sync) only when `failureCategory(for: error) == .accountChanged`; every other error (transient disk write — now surfaced by WC-M3's synchronous `update` — network blip, non-retryable record save, engine-state encode) is `report`-ed and the engine keeps running so CKSyncEngine retries.
    - **WC-L29 (`babf370`) — don't report "Up to Date" when a partial failure rejected records.** New static `partialFailureIsBenign`: `synchronize` reports `.upToDate` only when every per-item error is one the sent-side classifier already declines to throw on (retryable / `.serverRecordChanged` / `.zoneNotFound`) — kept in lock-step with `handleSentRecordZoneChanges` so the two paths never disagree. A genuine rejection (quota/permission/server-rejected/…) or an opaque (no-item) partial failure falls through to `report`. `isRetryable` made `static` to back the predicate.
    - **Tests** — `CloudSyncCoreTests` (offline, no live CloudKit): H3 batch-skip (one undecodable + one id-mismatch skipped, rest applied, no throw) + an all-bad batch; H4 `makeRecords` invokes the provider exactly once per batch (added a `ThreadSafeCounter` helper); L29 `partialFailureIsBenign` true for an all-benign partial failure, false for a `.quotaExceeded`/`.permissionFailure`/empty one. Updated `testRemoteUpsertAndDeleteRoundTrip` for the now-non-throwing `applyCloudSyncMutations`. M2/M3 are concurrency/semantics changes with no deterministic offline test — covered by the runtime checklist.
